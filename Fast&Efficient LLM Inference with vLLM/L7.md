# L7：Benchmarking and Evaluation 总结

本节主题是 **Measuring What Matters: Benchmarking and Evaluation**。前面的课程已经学习了模型量化和使用 vLLM 部署模型，本节进一步关注一个实际部署中更关键的问题：

> 这个 LLM 服务到底好不好？

这里的“好”不能只看模型能不能回答，也不能只看推理速度，而是要同时评估两个维度：

| 维度 | 核心问题 | 工具 |
|:--|:--|:--|
| Performance | 这个部署服务请求有多快、多稳定、多高效？ | GuideLLM |
| Quality | 这个模型在真实任务上的回答质量如何？ | lm_eval |

本节最终目标是：把性能 benchmark、任务质量评估和模型卡中的公开结果结合起来，判断一个量化模型是否值得部署。

---

## 1. 本节学习目标

本节主要学习四件事：

1. 使用 **GuideLLM** 对正在运行的 vLLM server 做真实性能 benchmark。
2. 解释 benchmark 中的关键性能指标，例如 TTFT、ITL、端到端延迟和吞吐。
3. 使用 **lm_eval** 在标准任务 Hellaswag 上评估模型质量。
4. 阅读量化模型的 model card benchmark 表格，理解量化带来的质量损失和 recovery。
5. 综合性能和质量证据，判断量化模型是否适合部署。

## 4. GuideLLM：部署性能 Benchmark 工具

[GuideLLM](https://github.com/neuralmagic/guidellm) 是 vLLM 项目中的 benchmark 工具，用来评估 LLM 推理服务的性能。它会以受控方式向推理服务发送请求，并记录每个请求的时间指标。

GuideLLM 主要回答的问题是：

> 这个 LLM deployment 的服务性能如何？

它关注的是部署层面的性能，而不是模型任务准确率。

### 4.1 关键性能指标

notebook 中介绍了几个重要指标：

| 指标 | 全称 | 含义 |
|:--|:--|:--|
| TTFT | Time To First Token | 从发送请求到收到第一个 token 的时间，反映用户感知响应速度 |
| ITL | Inter-Token Latency | token 与 token 之间的生成间隔，反映流式输出是否顺滑 |
| E2E latency | End-to-End Latency | 一个请求从开始到结束的总耗时 |
| Throughput | 吞吐 | 单位时间处理的请求数或生成 token 数 |

这些指标分别对应不同用户体验：

- **TTFT 低**：用户很快看到模型开始回答。
- **ITL 低**：流式输出更顺畅。
- **E2E latency 低**：整个回答完成更快。
- **Throughput 高**：系统整体服务能力更强。

## 5. 运行 GuideLLM Benchmark

notebook 使用 `subprocess.run` 调用命令行工具 `guidellm benchmark`：

```python
cmd = [
    "guidellm", "benchmark",
    "--target", "http://localhost:8000",
    "--model", MODEL,
    "--processor", "Qwen/Qwen3-0.6B",
    "--profile", "synchronous",
    "--max-requests", "10",
    "--data", "prompt_tokens=32,output_tokens=16,samples=32",
    "--output-dir", "./outputs",
]
```

参数解释：

| 参数 | 含义 |
|:--|:--|
| `guidellm benchmark` | 启动 GuideLLM benchmark |
| `--target http://localhost:8000` | 指定被测 vLLM server 地址 |
| `--model MODEL` | 指定服务端模型 ID |
| `--processor Qwen/Qwen3-0.6B` | 指定 tokenizer/processor，用于构造和统计请求 |
| `--profile synchronous` | 同步 profile，一次一个请求 |
| `--max-requests 10` | 最多发送 10 个请求 |
| `--data prompt_tokens=32,output_tokens=16,samples=32` | 构造输入/输出 token 规模 |
| `--output-dir ./outputs` | 保存 benchmark 结果 |

### 5.1 为什么只跑 10 个请求

notebook 中使用：

```text
--max-requests 10
```

这是为了让课程环境中运行更快。

真实生产 benchmark 通常应该使用：

- 几百到几千个请求
- 不同输入长度
- 不同输出长度
- 不同并发数
- 不同请求到达速率
- time-bounded benchmark，例如 `--max-seconds`

因此，本节的 benchmark 更偏教学演示，而不是完整生产压测。

## 6. 读取和解释 GuideLLM 结果

GuideLLM 会把结果保存到：

```text
outputs/benchmarks.json
```

notebook 读取该文件：

```python
with open("outputs/benchmarks.json") as f:
    report = json.load(f)

bench = report["benchmarks"][0]
metrics = bench["metrics"]
n_requests = metrics["request_totals"]["successful"]
```

然后提取并打印：

- TTFT
- ITL
- E2E Latency
- Output tokens
- requests/s
- output tokens/s

输出格式关注：

```text
Mean, p50, p95, p99
```

这说明 GuideLLM 不只是给一个平均值，而是提供每个指标的分布统计。

---

## 7. 为什么 Percentiles 很重要

notebook 特别强调：不能只看平均值。

例如：

```text
平均延迟 100ms，但 p99 延迟 2s
```

这意味着虽然大多数请求很快，但每 100 个用户中可能有 1 个用户会等很久。

常见 percentile 含义：

| Percentile | 含义 |
|:--|:--|
| p50 | 中位数，一半请求比它快，一半比它慢 |
| p95 | 95% 请求比它快，最慢的 5% 构成长尾 |
| p99 | 99% 请求比它快，最慢的 1% 是更极端的尾部 |

部署评估时尤其要看：

- p95 latency
- p99 latency
- p95 TTFT
- p99 TTFT

原因是线上系统的用户体验常常被尾延迟决定，而不是被平均值决定。

## 8. lm_eval：模型质量评估工具

GuideLLM 评估的是服务性能，但不能回答：

> 模型回答得准不准？

因此 notebook 使用 [lm_eval](https://github.com/EleutherAI/lm-evaluation-harness) 来评估模型质量。

`lm_eval` 用标准学术 benchmark 衡量模型在任务上的表现，例如：

- multiple-choice QA
- commonsense reasoning
- math
- factuality
- instruction following

本节使用的是：

```python
TASK = "hellaswag"
```

Hellaswag 是一个常识推理 benchmark，经常用于测试模型是否能选择最合理的句子续写。

---

## 9. GuideLLM 和 lm_eval 的区别

notebook 中给出了清晰对比：

| 对比项 | GuideLLM | lm_eval |
|:--|:--|:--|
| 衡量对象 | LLM 服务部署 | 模型任务能力 |
| 关注指标 | 延迟、吞吐、稳定性 | 准确率、任务分数 |
| 目标 | 这个 deployment 快不快 | 这个 model 答得好不好 |
| 输入对象 | 正在运行的推理 server | 本地模型或 API 模型 |

所以部署决策必须同时看两者：

- GuideLLM 好，只说明服务快。
- lm_eval 好，才说明模型质量可接受。

---

## 10. 使用 lm_eval 评估 vLLM Server

notebook 使用 `lm_eval.simple_evaluate`，并通过 OpenAI-compatible completions endpoint 调用 vLLM：

```python
results = lm_eval.simple_evaluate(
    model="local-completions",
    model_args=(
        f"model={MODEL},"
        f"base_url={VLLM_URL}/v1/completions,"
        "tokenized_requests=False,"
        "tokenizer=Qwen/Qwen3-0.6B,"
        "num_concurrent=1"
    ),
    tasks=[TASK],
    limit=20,
)
```

关键点：

| 参数 | 含义 |
|:--|:--|
| `model="local-completions"` | 使用本地 OpenAI-compatible completions API |
| `base_url={VLLM_URL}/v1/completions` | 指向 vLLM 的 completions endpoint |
| `tokenizer=Qwen/Qwen3-0.6B` | 使用 Qwen3-0.6B tokenizer |
| `num_concurrent=1` | 单并发评估，降低不确定性 |
| `tasks=["hellaswag"]` | 运行 Hellaswag |
| `limit=20` | 只跑 20 个样本，加快课程运行 |

### 10.1 为什么用 completions endpoint

Hellaswag 这类 multiple-choice benchmark 通常需要计算候选答案的 log probability。OpenAI-compatible 的 `completions` endpoint 更适合这种 logprob scoring，因此 notebook 使用：

```text
/v1/completions
```

而不是 chat completions。

### 10.2 为什么只跑 20 个 examples

`limit=20` 是为了快速演示。

真实评估时应该：

- 跑完整 test/validation set
- 多次重复评估
- 对关键任务使用更贴近业务的数据集
- 比较原始模型和量化模型

## 13. Recovery 指标的含义

模型卡中有一列：

```text
Recovery
```

它表示：

> 量化模型保留了原始模型多少比例的性能。

简单理解：

```text
Recovery = 量化模型分数 / 原始模型分数
```

例如 Hellaswag：

| 模型 | Hellaswag 分数 |
|:--|:--|
| Qwen3-0.6B | 43.04 |
| W4A16 | 41.02 |
| Recovery | 95.3% |

表示 W4A16 量化模型在 Hellaswag 上保留了约 95.3% 的原始性能。

## 14. Model Card 中的主要结果

notebook 中列出的部分关键结果：

### 14.1 OpenLLM v1

| Benchmark | 原始模型 | W4A16 | Recovery |
|:--|--:|--:|--:|
| MMLU | 42.82 | 39.80 | 93.0% |
| ARC Challenge | 32.85 | 30.72 | 93.5% |

| Winogrande | 54.54 | 54.62 | 100.1% |

### 14.2 OpenLLM v2 和 Reasoning

对于更难的任务，量化后的下降可能更明显，例如：

| Benchmark | 原始模型 | W4A16 |
|:--|--:|--:|
| IFEval | 62.83 | 55.81 |

| LiveCodeBench | 12.83 | 8.35 |

这说明量化并不是对所有能力影响都一样。某些复杂推理、数学、代码任务可能更敏感。

## 15. 如何判断量化模型是否值得部署

notebook 最后的核心问题是：

> 量化带来的性能收益，是否值得用质量损失来换？

部署决策需要综合三类证据：

| 来源 | 告诉你什么 |
|:--|:--|
| GuideLLM | 这个部署快不快、吞吐高不高、尾延迟是否可接受 |
| lm_eval | 模型在你运行的标准任务上回答质量如何 |
| Model card | 发布者在更多 benchmark 上的质量对比和 recovery |

不能只看一个维度：

- 如果模型吞吐翻倍，但准确率下降 15%，可能不值得。
- 如果模型很准确，但延迟无法满足 SLO，也不能上线。
- 如果模型大小减半，准确率只降 4%，可能是很好的部署折中。

notebook 对 W4A16 的判断是：

```text
约 50% 模型大小减少
约 4% OpenLLM v1 平均准确率损失
```

是否值得部署，取决于具体业务场景。

## 16. 实际部署时应该怎么评估

基于本节内容，实际部署量化模型时建议遵循以下流程：

1. **先测性能**
   - 使用 GuideLLM 或类似工具。
   - 测 TTFT、ITL、E2E latency、tokens/s、req/s。
   - 重点关注 p95/p99，而不仅是平均值。

2. **再测质量**
   - 使用 lm_eval 跑标准 benchmark。
   - 选择和业务相关的任务。
   - 不要只跑 20 个样本，生产评估应跑完整数据集。

3. **阅读 model card**
   - 查看发布者是否提供原始模型 vs 量化模型对比。
   - 关注 recovery。
   - 注意不同任务下降幅度可能不同。

4. **结合业务指标决策**
   - 如果业务是聊天助手，关注通用 QA、instruction following 和延迟。
   - 如果业务是数学/代码，特别关注 reasoning 和 coding benchmark。
   - 如果业务是高并发服务，关注吞吐和尾延迟。

5. **上线前做 A/B 或灰度测试**
   - benchmark 不能完全代表真实用户请求。
   - 最终还需要线上指标验证。

## 18. 核心结论

本节最重要的结论如下：

1. LLM 部署评估必须同时看 **performance** 和 **quality**。
2. GuideLLM 用于衡量服务性能，包括 TTFT、ITL、端到端延迟和吞吐。
3. 平均值不足以判断服务体验，p95 和 p99 尾延迟非常关键。
4. lm_eval 用于衡量模型在标准任务上的能力，例如 Hellaswag。
5. vLLM 的 OpenAI-compatible completions API 可以直接接入 lm_eval。
6. Model card 中的 accuracy table 可以帮助快速理解量化模型的整体质量损失。
7. Recovery 表示量化模型相对原始模型保留的性能比例。
8. W4A16 通常能显著减少模型大小，但不同任务上的质量下降幅度不同。
9. 是否部署量化模型取决于业务场景：延迟、吞吐、成本和准确率都要综合考虑。
