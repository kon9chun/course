最终目标是得到一个 `W4A16` 格式的量化模型。

## 2. 什么是 LLM Compressor

`llm-compressor` 是 vLLM 项目中的生产级量化工具。它可以在不重新训练模型的情况下，对已经训练好的模型进行低精度压缩。

它的核心 API 是：

```python
oneshot(
    model="model-name",
    dataset="dataset-name",
    recipe=recipe,
    output_dir="./output",
    num_calibration_samples=256,
    max_seq_length=4096,
)
```

`oneshot` 的含义是：

> 只需要对少量校准数据做一次处理，就能完成量化，不需要完整重新训练模型。

它需要三个关键输入：

| 输入 | 含义 |
|:--|:--|
| `model` | 要量化的模型，可以是 Hugging Face 模型 ID，也可以是本地路径 |
| `dataset` | 校准数据集，用来估计量化误差 |
| `recipe` | 量化 recipe，描述使用什么算法和量化配置 |

输出结果是一个压缩后的模型目录，可以被 vLLM 等推理框架直接加载。

## 4. Recipe：量化配置的核心

在 `llm-compressor` 中，**recipe** 用来描述如何量化模型。它可以由一个或多个 modifier 组成，每个 modifier 代表一种量化或变换策略。

### 4.1 Quantization Modifiers

notebook 中介绍了几类量化 modifier：

| Modifier | 作用 |
|:--|:--|
| `GPTQModifier` | 使用 GPTQ 算法，通过校准数据寻找更优量化权重 |
| `AWQModifier` | Activation-Weighted Quantization，保护对激活更重要的权重 |
| `AutoRoundModifier` | Intel 的量化方法，通过可学习 rounding/clipping 降低误差 |
| `QuantizationModifier` | 基础 PTQ/QAT 量化接口，适合简单场景 |

### 4.2 Transform Modifiers

除了直接量化，还有一些 transform modifier 用于改善量化前的数值分布，从而提升低精度模型质量。

| Modifier | 作用 |
|:--|:--|
| `SmoothQuantModifier` | 平滑激活分布，常用于改善 W8A8 量化效果 |
| `QuIPModifier` | 使用 Hadamard rotations 减少 outlier 影响 |
| `SpinQuantModifier` | 通过旋转使权重分布更均匀 |

这些 modifier 可以和量化 modifier 串联使用，例如先使用 `SmoothQuantModifier`，再使用 `GPTQModifier`。

---

## 5. W 和 A：量化方案的含义

量化 scheme 通常写作 `W?A?`：

- `W` 表示 **Weights**，即模型权重。
- `A` 表示 **Activations**，即推理时产生的中间激活。

notebook 中列出的常见 scheme：

| Scheme | 权重精度 | 激活精度 | 量化层压缩比例 | 质量影响 |
|:--|:--|:--|:--|:--|
| `W8A16` | 8-bit | 16-bit FP16/BF16 | 约 50% | 很小 |
| `W4A16` | 4-bit | 16-bit FP16/BF16 | 约 75% | 低到中等 |
| `W8A8` | 8-bit | 8-bit | 约 50% | 较低 |
| `W4A8` | 4-bit | 8-bit | 约 75% | 中等 |

本节采用的是：

```text
W4A16
```

含义是：

- Linear 层权重量化为 4-bit。
- 激活仍保持 16-bit。

因此，`W4A16` 更偏向于减少模型权重存储和显存占用，同时尽量保持推理质量。

## 6. 本节使用的 GPTQ W4A16 Recipe

notebook 中的 recipe 如下：

```python
from llmcompressor.modifiers.quantization import GPTQModifier

recipe = GPTQModifier(
    scheme="W4A16",
    targets="Linear",
    ignore=["lm_head"],
)
```

参数解释：

| 参数 | 值 | 含义 |
|:--|:--|:--|
| `scheme` | `W4A16` | 权重量化到 4-bit，激活保持 16-bit |
| `targets` | `Linear` | 只量化 Linear 层，因为这些层通常占据大部分参数 |
| `ignore` | `["lm_head"]` | 不量化输出层，避免影响词表 logits 的精度 |

这个配置的核心思想是：

> 量化模型中参数最多的 Linear 层，同时保留输出层精度，以取得压缩率和生成质量之间的平衡。

运行时 notebook 出现了如下 warning：

```text
triton not found; flop counting will not work for triton kernels
```

这只是提示当前环境没有 Triton，因此某些 FLOP 统计功能不可用。它不是 recipe 创建失败，也不影响这段 recipe 本身正常生成。

## 7. 为什么需要 Calibration Dataset

GPTQ 不是简单地把权重四舍五入到 INT4。它会使用一小批真实文本作为校准数据，估计量化后每个权重对模型输出的影响，然后寻找误差较小的量化表示。

校准数据的作用：

1. 给模型输入真实文本。
2. 观察模型在真实输入下的权重和激活行为。
3. 估计不同权重量化方式带来的输出误差。
4. 选择更合适的量化值，降低质量损失。

本节使用的数据集是：

```python
dataset="wikitext"
dataset_config_name="wikitext-2-raw-v1"
```

也就是 WikiText-2，它由 Wikipedia 文本组成，常用于语言模型评估。

notebook 中设置：

```python
num_calibration_samples=256
max_seq_length=4096
```

含义是使用 256 个校准样本，每个样本最大序列长度为 4096 tokens。

## 8. 执行量化：oneshot

量化代码如下：

```python
from llmcompressor import oneshot

if not os.path.isdir(OUTPUT_DIR):
    oneshot(
        model=MODEL_DIR,
        dataset="wikitext",
        dataset_config_name="wikitext-2-raw-v1",
        recipe=recipe,
        output_dir=OUTPUT_DIR,
        max_seq_length=4096,
        num_calibration_samples=256,
    )
    print(f"Quantization complete. Model saved to: {OUTPUT_DIR}")
```

这里有一个重要判断：

```python
if not os.path.isdir(OUTPUT_DIR):
```

它的作用是：

- 如果量化模型目录已经存在，就跳过量化。
- 如果量化模型目录不存在，就运行 `oneshot` 重新量化。

这可以避免每次运行 notebook 都重新量化模型，因为量化过程可能需要几分钟，并且对 GPU/内存有一定要求。

本 notebook 当前使用的是本地模型路径：

```python
model=MODEL_DIR
```

这样比在线写成 `model="Qwen/Qwen3-0.6B"` 更稳定，因为它直接使用本地已经下载好的模型。

## 9. 模型大小对比

notebook 定义了两个函数统计目录大小：

```python
def folder_size(path):
    p = pathlib.Path(path)
    if not p.exists():
        return 0
    return sum(f.stat().st_size for f in p.rglob("*") if f.is_file())

def format_size(nbytes):
    if nbytes < 1024**2:
        return f"{nbytes/1024:.1f} KB"
    if nbytes < 1024**3:
        return f"{nbytes/1024**2:.1f} MB"
    return f"{nbytes/1024**3:.2f} GB"
```

然后比较原始模型和量化模型大小：

```python
size_orig = folder_size(MODEL_DIR)
size_q = folder_size(OUTPUT_DIR)
reduction = (1 - size_q / size_orig) * 100
```

本次运行结果是：

| 模型 | 大小 |
|:--|:--|
| 原始 BF16 模型 | 1.41 GB |
| W4A16 量化模型 | 528.7 MB |
| 压缩比例 | 64% |

也就是说，量化后模型从 1.41 GB 降到 528.7 MB，体积减少约 64%。

需要注意：理论上从 16-bit 到 4-bit 是 4 倍压缩，也就是量化层可减少约 75%。但整个模型的实际压缩比例通常低于理论值，因为并不是所有部分都会被量化。

未量化或高精度保留的部分包括：

- embedding
- `lm_head`
- normalization 层
- 一些 metadata 和 tokenizer 文件

因此，总体压缩比例取决于被量化层在整个模型中的占比。

## 11. Perplexity：用困惑度评估质量损失

Perplexity 可以理解为：

## 12. Perplexity 计算方法

notebook 定义了：

```python
def calculate_perplexity(model, tokenizer, dataset, max_tokens=5000, stride=512):
    encodings = tokenizer(
        "\n\n".join(dataset["text"]),
        return_tensors="pt",
        truncation=True,
        max_length=max_tokens,
    )
    input_ids = encodings.input_ids
    nlls, prev_end = [], 0

    for begin_loc in range(0, input_ids.size(1), stride):
        end_loc = min(begin_loc + stride, input_ids.size(1))
        trg_len = end_loc - prev_end
        input_slice = input_ids[:, begin_loc:end_loc]
        target_slice = input_slice.clone()
        target_slice[:, :-trg_len] = -100
        with torch.no_grad():
            loss = model(input_slice, labels=target_slice).loss
            nlls.append(loss * trg_len)
        prev_end = end_loc

    return math.exp(torch.stack(nlls).sum() / prev_end)
```

逻辑如下：

1. 把 WikiText-2 的文本拼接起来。
2. 使用 tokenizer 编码为 token IDs。
3. 按 `stride=512` 切分输入，避免一次处理太长文本。
4. 计算每段文本的 negative log likelihood。
5. 加权平均 loss。
6. 对平均 loss 取指数，得到 perplexity。

公式上可以理解为：

```text
perplexity = exp(average negative log likelihood)
```

## 14. 本节中的关键 trade-off

本节展示的是 LLM 量化中的典型权衡：

```text
更小模型 / 更低显存 / 更低部署成本
                vs
更高量化误差 / perplexity 上升 / 生成质量可能下降
```

对于 `GPTQ + W4A16`：

优点：

- 模型文件显著变小。
- 权重从 16-bit 降到 4-bit。
- 不需要重新训练。
- 可直接用于 vLLM 等推理引擎。
- 质量下降通常比较温和。

缺点：

- 量化过程需要校准数据。
- 量化模型 perplexity 会升高。
- 某些 runtime 对 W4A16 量化格式可能需要 GPU 支持。
- 并非所有层都量化，因此实际压缩率低于理论 4 倍。

---

## 15. W4A16 为什么常用

`W4A16` 是 LLM 部署中常见的折中方案。

它的特点是：

- 权重使用 4-bit，显著减少模型大小。
- 激活仍保持 16-bit，降低激活量化带来的误差。
- 比 `W4A8` 更稳。
- 比 `W8A16` 压缩更强。
- 适合对显存敏感、但仍希望保持生成质量的场景。

在本节中，`W4A16` 的实际结果是：

```text
模型大小：1.41 GB -> 528.7 MB
perplexity：32.80 -> 35.18
```

这正体现了它的价值：用一定质量损失换取显著压缩。

## 17. 核心结论

本节最重要的结论如下：

1. `llm-compressor` 可以使用 `oneshot` 对模型做训练后量化，不需要重新训练。
2. GPTQ 通过校准数据优化量化权重，比简单四舍五入更准确。
3. `W4A16` 表示权重 4-bit、激活 16-bit，是一种常见的 LLM 压缩方案。
4. 量化主要针对 Linear 层，因为这些层占据模型大部分参数。
5. `lm_head` 通常保留高精度，以避免输出词表映射精度下降。
6. 本例中模型大小从 1.41 GB 降到 528.7 MB，减少约 64%。
7. 量化后 perplexity 从 32.80 升到 35.18，增加约 7.2%。
8. 量化模型生成结果仍然合理，说明质量损失可控。
9. 量化的核心价值是降低模型部署成本，但需要接受一定精度损失。
