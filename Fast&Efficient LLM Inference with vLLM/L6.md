# L6：使用 vLLM 高效部署和服务 LLM 总结

本节主题是 **Serving LLMs Efficiently with vLLM - Part II**，重点从上一节模型量化过渡到模型服务：如何使用 vLLM 启动推理服务，通过 OpenAI 兼容 API 调用模型，并观察 vLLM 内部的关键优化机制，例如连续批处理、PagedAttention、KV Cache、Prefix Caching 和 Thinking Mode。

## 1. 从量化到服务

前面的课程使用 `llm-compressor` 对模型权重进行 GPTQ 量化，把权重从 16-bit 压缩到 4-bit。L6 则关注另一个阶段：**如何把模型高效地部署成可访问的推理服务**。

[vLLM](https://github.com/vllm-project/vllm) 是一个开源 LLM 推理引擎，主要目标是提高大语言模型的吞吐、显存利用率和服务效率。

vLLM 的关键能力包括：

| 功能 | 作用 |
|:--|:--|
| Continuous batching | 在 token 级别动态调度请求，避免静态 batch 中等待最长请求造成的计算浪费 |
| PagedAttention | 把 KV cache 切分成固定大小的 block 管理，减少显存碎片和浪费 |
| Prefix caching | 对多个请求共享的 prompt 前缀复用 KV cache，减少重复 prefill 计算 |
| Quantization support | 支持 GPTQ、AWQ、compressed-tensors 等量化模型格式 |
| OpenAI-compatible API | 提供与 OpenAI SDK 兼容的接口，便于迁移已有应用 |

## 2. 启动 vLLM 服务

notebook 中假设学习环境已经预热并启动了一个 vLLM server，服务的模型是 Qwen3-0.6B。实际部署时，可以在终端运行类似命令：

```bash
vllm serve Qwen/Qwen3-0.6B --dtype=bfloat16 --max-model-len 4096
```

参数含义：

| 参数 | 含义 |
|:--|:--|
| `vllm serve` | 启动 vLLM 内置 HTTP 推理服务器 |
| `Qwen/Qwen3-0.6B` | Hugging Face 上的模型 ID，首次运行会自动下载模型和 tokenizer |
| `--dtype=bfloat16` | 使用 BF16 精度加载模型权重 |
| `--max-model-len 4096` | 限制最大上下文长度为 4096 tokens |

vLLM server 默认通过 HTTP 暴露接口，常见地址是：

```python
VLLM_URL = "http://localhost:8000"
```

notebook 通过访问：

```text
/v1/models
```

检查服务是否已经启动，并读取当前服务模型的 ID。

## 3. OpenAI 兼容 API

vLLM 的一个重要优势是提供 **OpenAI-compatible API**。这意味着可以直接使用 OpenAI Python SDK 调用本地 vLLM 服务：

```python
from openai import OpenAI

client = OpenAI(
    base_url="http://localhost:8000/v1",
    api_key="unused",
)
```

这里 `api_key="unused"` 是因为本地 vLLM 服务一般不需要真正的 OpenAI API key，只是 SDK 参数要求传入。

之后可以像调用 OpenAI Chat Completions 一样调用本地模型：

```python
resp = client.chat.completions.create(
    model=MODEL,
    messages=[{"role": "user", "content": "What is PagedAttention in one sentence?"}],
    max_tokens=80,
    temperature=0.7,
    top_p=0.8,
    extra_body={
        "top_k": 20,
        "chat_template_kwargs": {"enable_thinking": False},
    },
)
```

## 4. Qwen3 的 Thinking Mode

Qwen3 支持一种 **thinking mode**，模型会先生成内部推理内容，再输出最终回答。notebook 中为了让回答更短、更稳定，通常关闭 thinking：

```python
extra_body={
    "chat_template_kwargs": {"enable_thinking": False}
}
```

如果开启 thinking mode，模型可能生成 `<think>...</think>` 形式的中间推理内容。它可能提高回答质量，但也会带来：

- 更多生成 token
- 更高计算量
- 更大的 KV cache 占用
- 更长响应时间

因此在服务场景中，是否开启 thinking mode 是质量、延迟和成本之间的权衡。

## 5. 使用 Logprobs 观察模型行为

notebook 展示了如何通过 `logprobs=True` 查看模型每个输出 token 的概率信息。

示例：

```python
resp = client.chat.completions.create(
    model=MODEL,
    messages=[{"role": "user", "content": "The capital of France is"}],
    max_tokens=15,
    temperature=0.0,
    logprobs=True,
    top_logprobs=5,
    extra_body={"chat_template_kwargs": {"enable_thinking": False}},
)
```

关键参数：

| 参数 | 作用 |
|:--|:--|
| `temperature=0.0` | 使用确定性输出，更容易分析概率 |
| `logprobs=True` | 返回每个生成 token 的 log probability |
| `top_logprobs=5` | 返回候选概率最高的 5 个 token |

logprob 是对数概率。可以用：

```python
math.exp(logprob)
```

转换为普通概率。

通过 logprobs 可以观察：

- 模型选择某个 token 的置信度
- 模型还考虑了哪些替代 token
- 生成过程是否存在明显不确定性
- 不同采样参数如何改变输出分布

## 6. Prometheus Metrics：观察 vLLM 内部状态

vLLM 提供 Prometheus 兼容的 `/metrics` endpoint，可以观察服务端内部指标：

```text
http://localhost:8000/metrics
```

notebook 中定义了函数：

```python
def get_vllm_metrics(base_url=VLLM_URL):
    r = requests.get(f"{base_url}/metrics")
    metrics = {}
    for line in r.text.split("\n"):
        if line.startswith("#") or not line.strip():
            continue
        name = line.split("{")[0].split()[0]
        try:
            metrics[name] = float(line.split()[-1])
        except (ValueError, IndexError):
            continue
    return metrics
```

重点观察的指标包括：

| 指标 | 含义 |
|:--|:--|
| `vllm:num_requests_running` | 当前正在运行的请求数 |
| `vllm:num_requests_waiting` | 当前排队等待的请求数 |
| `vllm:gpu_cache_usage_perc` | GPU KV cache 使用比例 |
| `vllm:cpu_cache_usage_perc` | CPU KV cache 使用比例 |
| `vllm:prompt_tokens_total` | 累计处理的 prompt tokens 数 |
| `vllm:generation_tokens_total` | 累计生成 tokens 数 |
| `vllm:prefix_cache_queries_total` | prefix cache 查询次数 |

这些指标对于部署 LLM 服务非常重要，可以用于判断：

- 服务是否拥塞
- KV cache 是否接近上限
- 请求是否排队
- 生成吞吐是否正常
- prefix caching 是否生效

## 7. Continuous Batching：连续批处理

传统静态 batching 通常需要等一个 batch 内所有请求完成后，才能开始下一批。问题是不同请求生成长度不同，短请求完成后仍要等待长请求，造成 GPU 计算资源浪费。

vLLM 使用 **continuous batching**，也叫 iteration-level scheduling。它在每个生成 step 动态调度请求：

1. 多个请求一起进入 batch。
2. 每一步为 batch 中的请求各生成一个 token。
3. 某个请求生成完成后，立即释放 slot。
4. 新请求可以马上补进来，不需要等整个 batch 完成。

这带来的好处是：

- 提高 GPU 利用率
- 减少空等
- 提升吞吐
- 更适合在线服务中的混合长度请求

notebook 中通过 `ThreadPoolExecutor` 并发发送 5 个请求：

```python
with concurrent.futures.ThreadPoolExecutor(max_workers=len(prompts)) as pool:
    futures = {pool.submit(_ask, p): p for p in prompts}
```

然后通过 metrics 查看：

```text
num_requests_running
num_requests_waiting
```

以观察多个请求是否被同时调度。

需要注意：notebook 只在请求发出后很短时间采样一次 metrics，因此第一次运行可能看到 `running: 0` 或 `running: 1`，这不一定代表 continuous batching 没生效，只是采样时机过早。重复运行或连续采样更能看到真实峰值。

## 8. PagedAttention 与 KV Cache 管理

LLM 自回归生成时，每生成一个 token 都需要使用前面 token 的 attention 信息。为了避免重复计算历史 token，模型会保存每层 attention 的 Key 和 Value，这就是 **KV cache**。

KV cache 的特点：

- 随上下文长度线性增长
- 随并发请求数线性增长
- 长上下文和高并发时占用大量显存
- 是 LLM serving 的核心内存瓶颈之一

vLLM 的 **PagedAttention** 借鉴操作系统分页思想，把 KV cache 切成固定大小的 blocks：

- 每个请求的 KV cache 不必连续存放。
- block 可以分散在显存中。
- 请求结束后，block 可以立刻回收。
- 减少显存碎片。
- 提高 KV cache 利用率。

这也是 vLLM 能高效支持 continuous batching 的关键基础。

## 9. Prefix Caching：前缀缓存

很多生产应用会有共享的长 system prompt，例如：

- 企业助手的统一角色设定
- RAG 系统的固定指令
- few-shot examples
- 工具调用说明
- 安全策略说明

如果每个请求都重新计算这部分前缀，会浪费大量 prefill 计算。

vLLM 的 **prefix caching** 会检测多个请求中相同的 prompt 前缀，并复用已计算好的 KV cache。

工作方式可以理解为：

1. 第一个请求包含长 system prompt。
2. vLLM 为这个前缀计算 KV cache。
3. 后续请求如果拥有相同前缀，就复用已有 KV cache。
4. 后续请求只需要计算不同的用户问题部分。

notebook 中连续发送 5 个请求，它们共享同一个 `SYSTEM_PROMPT`，然后观察：

```text
vllm:prefix_cache_queries_total
```

如果该指标增加，说明 vLLM 正在检查并尝试复用 prefix cache。

在 notebook 示例里 system prompt 较短，所以性能收益可能不明显。但在生产中，如果 system prompt 有几千 tokens，prefix caching 可以显著减少 prefill 时间和显存/计算开销。

## 10. KV Cache 大小估算

notebook 使用 Qwen3-0.6B 的配置估算 KV cache 占用：

```python
num_layers = 28
num_kv_heads = 8
head_dim = 128
dtype_bytes = 2  # BF16

per_token = 2 * num_layers * num_kv_heads * head_dim * dtype_bytes
```

公式含义：

```text
每 token KV cache = 2 × 层数 × KV heads 数 × head_dim × 每个元素字节数
```

其中第一个 `2` 表示同时保存：

- Key
- Value

对 Qwen3-0.6B：

```text
2 × 28 × 8 × 128 × 2 = 114,688 bytes ≈ 112 KB/token
```

所以：

| 上下文长度 | KV Cache 大小 |
|:--|:--|
| 1 token | 约 112 KB |
| 64 tokens | 约 7 MB |
| 256 tokens | 约 28 MB |
| 1024 tokens | 约 112 MB |
| 4096 tokens | 约 448 MB |

如果有 10 个并发请求，每个 4096 tokens：

```text
约 4.4 GB KV cache
```

这说明 KV cache 是长上下文和高并发场景下非常重要的内存开销。

## 11. Thinking Mode 的服务成本

notebook 最后对比了 Qwen3 的 thinking mode 开启和关闭：

```python
extra_body={"chat_template_kwargs": {"enable_thinking": thinking}}
```

关闭 thinking：

- 回答更短
- 速度更快
- token 数更少
- KV cache 和计算开销更低

开启 thinking：

- 生成中间推理内容
- 可能提升复杂问题回答质量
- 生成 token 明显增加
- 延迟和成本更高

这说明在实际部署中，不能只看回答质量，还要考虑：

- 用户可接受延迟
- 单次请求成本
- token 预算
- KV cache 压力
- 吞吐需求

## 13. 核心结论

本节最重要的结论是：

1. vLLM 不只是一个模型加载工具，而是面向在线推理服务的高性能引擎。
2. OpenAI-compatible API 让本地模型服务可以无缝接入已有 OpenAI SDK 应用。
3. Continuous batching 解决了在线请求长度不一致导致的 GPU 低利用率问题。
4. PagedAttention 通过块化管理 KV cache，大幅降低显存碎片和浪费。
5. Prefix caching 可以复用共享 prompt 的 KV cache，尤其适合长 system prompt 和 few-shot 场景。
6. `/metrics` endpoint 是观察和调试 vLLM 服务状态的重要工具。
7. KV cache 随上下文长度和并发请求线性增长，是 LLM serving 的关键资源瓶颈。
8. Thinking mode 会消耗更多 tokens 和 KV cache，需要根据质量和成本权衡是否开启。
